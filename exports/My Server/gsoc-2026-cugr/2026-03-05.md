### 06:01 - @Soumyadip

Nice  <@562957765312643083> ! I've been analyzing this exact problem. Keeping the pre-ripup snapshot is dangerous because after ripping dirty nets, graph_edges_ demand decreases locally, making some congested regions appear artificially free  dirty nets could route through areas still congested by non-dirty nets.
Full re-extraction is expensive though. I think a smarter approach is partial re-extraction scoped to the combined bounding box of all dirty nets, while keeping the pre-ripup snapshot outside that region. This preserves routing stability for untouched nets while giving dirty nets accurate congestion data.
One thing I noticed  patternRouteWithDetours weights congestion more heavily in its detour cost function compared to patternRoute, so the staleness problem is actually worse in that stage specifically.
<@1478370910749855888>  does the bounding box approach make sense, or would it create boundary artifacts at the edges?

### 12:45 - @Soumyadip

<@562957765312643083>  Following up on the architectural review of your draft.

I just pushed PR #9647 to handle the actual GridGraph state invalidation.

As it stands, your incremental API is going to hit severe phantom congestion during the rip-up phase because the underlying 3D edge capacities aren't being explicitly released back to the router. You cannot reliably trigger a reroute without mathematically clearing the historical grid state first.

I've exposed grid_graph_->clearNetCapacity(net_id) in the core. You will need to inject that hook directly before your rerouteNets() call to ensure the grid space is actually freed and memory-safe.

Review the implementation when you have a minute, and let's get your API properly synchronized with the engine before we push it further.

### 12:47 - @Soumyadip

Just a quick follow-up to the grt updates I shared earlier. I’ve just pushed PR #9647 which implements the core state invalidation logic inside the GridGraph.
While the high-level incremental API is being explored, I wanted to ensure the engine has a mathematically sound way to handle the rip-up phase.
