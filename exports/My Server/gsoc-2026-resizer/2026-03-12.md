Hi , <@1478367235327266919> , <@1478321901993459765> Respected Mentors, I was looking into how the Resizer handles buffer insertion for long wires and max_tran/max_cap violations. I know standard EDA tools use a variant of van Ginneken’s dynamic programming algorithm, building a topological tree of candidate buffer solutions from the sinks back to the root, keeping a Pareto frontier of (Required Time, Capacitance) pairs.In modern Non-Linear Delay Models (NLDM), the delay of a cell isn't linearso it's heavily dependent on the input transition time (slew). The strict 'Principle of Optimality' in dynamic programming breaks down. A candidate buffer option that looks strictly worse right now (worse Required Time, worse Capacitance) might actually have a much sharper input slew. That sharper slew could make the upstream driver so much faster that this 'sub-optimal' option actually becomes the globally optimal path.
How does OpenROAD’s rsz engine handle this slew-induced non-monotonicity? Does it introduce 'Slew' as a third dimension in the dynamic programming space ,which would cause a massive combinatorial explosion in runtime or does it use a slew-bounding heuristic to aggressively prune the tree without accidentally killing the globally optimal solution?
